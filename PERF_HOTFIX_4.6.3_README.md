# 🚀 Hotfix 4.6.3 — connection_limit 1 → 3 (стабильность под нагрузкой)

## Что чинит

В логах из 15:41 видно главное узкое место:

| Что | Цифра |
|-----|-------|
| `/api/runs/start` (фоновая генерация) висит | **144 секунды** |
| Все другие запросы умирают через | **20 секунд** ровно (pool_timeout) |
| Ошибки 500 на `/workflows/...` за этот период | **22 раза** |
| Ошибки 500 на `/dashboard` за этот период | **4 раза** |

**Виновник:** `connection_limit=1` в DATABASE_URL. Когда executor через `after()` запускает фоновую генерацию (которая ждёт fal.ai 1-3 минуты), он держит **единственное** соединение к БД. Все остальные запросы (dashboard, переход на другой workflow, polling) встают в очередь на 20с → таймаут → 500.

**Фикс:** поднимаю `connection_limit` с **1** до **3**. Это значит что одновременно может быть 3 prisma-клиента на лямбду — фоновая генерация и dashboard перестанут друг друга блокировать.

## Почему это безопасно

Supabase официальная дока буквально говорит: *"set connection_limit=1 and gradually increase if necessary"*. Necessary — сейчас. У нас есть `after()` background work которого нет в типичном serverless паттерне, поэтому ставить 1 нельзя.

Supavisor (pgBouncer) **всё ещё мультиплексит** соединения server-side — мы не открываем 3 реальных Postgres сессии, просто разрешаем себе 3 client-side слота в pooler.

Также сократил `pool_timeout` с 20с до 5с — если 3 не хватит, пользователь быстро увидит 500 и сделает retry, вместо 20-секундного белого экрана.

## Тест

1. Hard refresh (Cmd+Shift+R)
2. Запусти долгую генерацию (Video Generation, 30-60s)
3. **Сразу** уйди на /dashboard
4. ✅ Dashboard должен открыться нормально, не за 20с и без 500

Если у тебя одновременно идут 2 видео-генерации параллельно — проверь что dashboard всё равно открывается. Это уже хороший stress-test для нового лимита.

## Что НЕ исправит этот хотфикс

- **Холодные старты Vercel Hobby** (2-5с на первую загрузку) — лечится только Vercel Pro
- **Supabase Free statement_timeout = 8s** — лечится только Supabase Pro + поднятием лимита через SQL
- **Supabase Free auto-pause через 7 дней** — лечится только Supabase Pro

Если хочешь "в несколько раз быстрее" — нужен апгрейд тарифов, отдельным шагом. См. предыдущий ответ.
